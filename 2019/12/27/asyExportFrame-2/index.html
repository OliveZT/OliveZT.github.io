二. 伪代码实现

首先还是祭出上篇博客中的架构图：

7. createExportTask

// 框架提供的远程rpc接口
@Autowired
private ExportTaskOperateService exportTaskOperateService;

@RequestMapping("client/createExportTask")
    public String createExportTask(@RequestBody ClientExportQueryDTO clientExportQueryDTO) {
        try {
            ExportTaskReqDTO exportTaskReqDTO = new ExportTaskReqDTO();
            exportTaskReqDTO.setRule(JSON.toJSONString(clientExportQueryDTO));
            // 设置导出的业务类型
            exportTaskReqDTO.setExportType(type);
            exportTaskReqDTO.setFormName("xxx");
            // 从session获取操作人ID，用于后续的权限控制
            Long userId = SessionContextHolder.getUserId();
            exportTaskReqDTO.setUserId(userId);
            Response<Boolean> response = exportTaskOperateService.createExportTask(exportTaskReqDTO);
            if (response != null && response.isSuccess()) {
                return response.getData().toString();
            }
        } catch (Exception e) {
            // 异常处理
        }
        return "fail";
    }

三. 关于后续优化

这周有同学过来和我对接异步下载框架。在他的项目中有超过10个地方需要使用Excel的导出功能。而在我框架设计的时候，每一个Excel的导出类型都需要一个定时任务去执行的。如果在这样的场景下，就需要建立超过10个定时任务去进行执行。虽然开发量很少，但是过多的定时任务如何合理的监管也是一个大问题；并且重复的类似代码也不够优雅（作为程序员真的要时刻保持优雅呢）。如何以更优雅更人性化的方式去处理这个问题，将是我之后的一个优化点。
