中间件相关知识

一. 背景

中间件相关知识～主要有Dubbo、MQ以及Redis

二. 开整！

2.1. Dubbo（Tesla）

1、rpc服务暴露过程
用户实现自定义接口服务，Dubbo会在Spring实例化完bean后，将具体服务转化Invoker；调用export方法，将Invoker转化为Exporter，将服务注册到服务中心。

2、rpc服务消费过程
客户端在本地配置（xml或者注解）了远程服务，通过调用本地方法一样去调用远程方法，rpc框架会创建invoker以及接口的代理，实际的调用是委托给代理rpcproxy。（这里做了多个服务提供者的合并，负载均衡等）

3、请求-响应过程（有些问题）
rpc服务方通过rpcServer去导出（export）远程接口方法，客户端通过rpcClient引入（import）远程接口方法。客户端通过调用本地方法一样去调用远程方法，rpc框架提供接口的代理实现，实际的调用是委托给代理rpcProxy。代理封装调用信息并将调用转交给rpcInvoker去执行。客户端的rpcInvoker通过连接器rpcConnector去维持与服务端的通道（rpcChannel），并用rpcProtocol执行协议编码（encode）并将编码后的请求消息通过通道发送给服务方。

rpc服务端接收器rpcAcceptor接收客户端的调用请求，同样使用rpcProtocol执行协议解码（decode）。解码后的调用信息传递给rpcProcessor去控制处理调用过程，最后rpcInvoker委托调用执行并返回调用结果。

4、Dubbo中的几个角色
1）Provider：暴露服务的服务提供方；
2）Consumer：调用远程服务的服务消费方；
3）Registry：服务注册与发现的注册中心；
4）Monitor：服务消费者和提供者在内存中累计调用次数和调用时间，主动定时每分钟发送一次统计数据到监控中心；
5）Container：服务运行容器，远程调用、序列化

4、Dubbo中的负载均衡
1）随机，按权重随机选择，也是默认的策略；
2）轮询；
3）最少活跃调用；
4）一致性哈希，相同参数请求总是发到同一个provider

5、集群容错策略
1）Failover Cluster：调用失败后自动切换重新调用其他机器，适合读操作，默认模式；
2）Failback Cluster：调用失败后，返回一个空结果给服务提供者，通过定时任务重发，适合执行消息通知；
3）Failfast Cluster：一次调用失败后抛出异常，适合保持幂等性的操作，如重复提交订单；
4）Failsafe Cluster：调用失败后不抛出异常，只打印异常信息，适用于日志审计；
5）Forking Cluster：多线程并发地去调用，如果有一个返回成功了，就会结束其他调用，这个比较适合实时性要求高的读操作；
6）Broadcast Cluster：逐个调用所有的provider，常用于通知所有提供者更新缓存或日志等本地资源信息。

6、什么是分布式事务，怎么实现
分布式事务简单来讲就是一个服务调用涉及到多处的资源协调，如果要整个服务做事务，就涉及到了多个分布式服务之间的事务协调。可以通过消息机制异步对账，保证最终的一致性。下单资源同步扣减资源，最终消息报保证一致。

2.2. MQ （RabbitMQ）

1、为什么使用消息队列
异步解耦、削峰填谷

2、消息队列如何选型
RocketMQ和Kafka吞吐量较高，达10万级；
RabbitMQ的一大特点，延迟最低，可达微秒级；
Kafka适合日志采集。

3、如何保证消息中间件的高可用
RabbitMQ 有三种模式：单机模式、普通集群模式（创建的queue，只会放在一个RabbitMQ实例上）、镜像集群模式（无论元数据还是queue 里的消息都会存在于多个实例上）。

Kafka 的高可用性：Kafka由多个 broker 组成，每个 broker 是一个节点；你创建一个 topic，这个 topic 可以划分为多个 partition，每个 partition 可以存在于不同的 broker 上，每个 partition 就放一部分数据。这是天然的分布式消息队列，就是说一个 topic 的数据，是分散放在多个机器上的，每个机器就放一部分数据。

4、如何保证消息不丢失
RabbitMQ：分三种情况：生产者弄丢了数据、RabbitMQ弄丢了数据以及消费者弄丢了数据。
1）生产者弄丢了数据：开启confirm模式，如果消息成功写入了RabbitMQ中，RabbitMQ会给你回传一个ack消息。否则认为消息没有成功发送，我们可以重发；
2）RabbitMQ 弄丢了数据：开启 RabbitMQ 的持久化，将消息持久化到磁盘；
3）消费者弄丢了数据：使用RabbitMQ提供的ack机制，关闭自动ack。在客户端处理完消息的时候，在程序里 通过ack告知RabbitMQ才算消费成功。否则

Kafka：Kafka可以在producer端设置 acks=all，当leader接收到消息，所有的 follower都同步到了消息之后，才认为本次写成功了。如果没满足这个条件，生产者会自动不断的重试；
另外一种情况是：Kafka 的 leader 机器宕机了，在将follower切换为 leader 之后，会发现数据丢失。怎么解决这种情况呢？除了上面说的acks=all外，通过设置一些参数，保证每个 partition 必须有至少 2 个副本，要求一个leader至少有至少一个follower还处于存活状态。

5、如何保证消息不被重复消费
消息中间件基本都会有一个偏移量来保证消息消费的进度，如果当consumer已经消费了消息但是还没有将偏移量告知给消息中间件时，消息中间件进程crash了，此时认为consumer并没有消费该消费，重启后就会出现消息重复消费的情况。

一般来说，如果要保证消息不丢，就没有办法保证消息不被重复消费。没有办法两全其美。所以在业务上要保证消息的幂等。比如物流发货，物流单号和公司是一个唯一键约束，可以保证幂等；某些对于状态实效性要求不高的系统，如品控，可以通过异步对账保证最终一致性；插入db前根据是否存在此记录来决定是update还是insert。

5、如何保证消息传输的顺序性
Kafka和蘑菇街的Corgi在创建topic都有partition（分区）的概念。为了保证消息传输的顺序性，可以用某个字段作为分区字段比如退款的id，同一个退款id的消息会被放入一个partition中，所以一个partition里的数据是有序的。

6、消息积压了怎么办
一般对于消息的积压，我们都会有报警，如果consumer积压的消息超过阈值就会告警提醒关注者去处理。一般出现这种情况是因为消息在消费时，出现异常导致一直阻塞消费导致的，此时就需要去人工处理问题，然后让消息正常消费即可。

case：物流订阅消息不符合物流的规则校验导致业务异常，消息阻塞。物流在订阅时都会对物流公司编码及单号做一层简单的校验。比如某个物流公司的单号是纯数字12-16位，当时是有个业务入口层没有调用物流的单号校验导致这种异常的情况发生

7、蘑菇街消息中间件-Corgi架构
采用RocketMQ+Kafka的架构：Keeper（有点类似于redis的哨兵，在RocketMQ 中叫做NameServer；Kafka使用ZooKeeper做元数据的存储及Broker的发现）采用主从的架构，提供服务发现和元数据管理的功能；Broker之间采用主从的结构。

2.3. ZK

1、zk是什么
ZooKeeper是一个经典的分布式数据一致性解决方案，致力于为分布式应用提供一个高性能、高可用，且具有严格顺序访问控制能力的分布式协调服务。zk的数据结构是一个树形结构

2、zk的使用
分布式应用程序可以基于ZooKeeper实现数据发布与订阅、负载均衡、命名服务、分布式协调与通知、集群管理、Leader选举、分布式锁、分布式队列等功能。

3、业务使用：
rpc注册中心。

4、zk集群角色
ZooKeeper没有沿用传统的Master/Slave模式（主备模式），而是引入了Leader、Follower和Observer三种角色。Observer和Follower唯一的区别是Observer不参与任何形式的投票。

2.4. Redis

1、Redis的六种底层数据结构
简单动态字符串（SDS）、链表、字典、跳跃表，整数集合和压缩列表

2、简单动态字符串（SDS）
SDS的结构体中维护了一个字节数组buf[]用来保存字符串，len表示现有字符串占用的空间，free表示剩余可用的空间长度。

相比于string的区别
（1）SDS中获取字符串的长度的时间复杂度是O(1)；
（2）自动空间预分配，不用人工分配空间，并减少了由于修改字符串长度导致了空间分配次数（小于1MB，分配两倍所需空间，否则长度加1MB）；也不会造成缓冲区溢出
（3）SDS不是用空字符串来判定字符串结尾的，因此可以保存二进制数据；
（4）SDS也有一定的局限，SDS只能用一部分<string.h>库的函数

3、链表（list）结构
Redis列表（List）底层实现之一就是链表。list结构体内保存着链表节点数目的属性len，那么我们在获取链表长度的时候就不需要一直迭代了，时间复杂度只有O(1)；并且我们可以在O(1)的时间复杂度内获取到链表的头结点或者尾节点。

4、字典（dictionary）结构
Redis的哈希对象的底层就是通过字典来实现的，Redis底层默认使用MurmurHash2算法来计算哈希值；rehash（再散列）的长度计算是计算第一个大于等于size的2次幂的数；

字典有两个hash表，如果0号哈希表为空，那么将新哈希表设置为0号哈希表，那么此次操作其实就是一次初始化；否则，将新哈希表设置为1号哈希表，这个过程就是一次rehash。

Redis使用了渐进式的rehash策略——rehash是多次、渐进地完成的：有一个rehashidx字段来控制rehash的进度。

5、跳跃表（skiplist）
对排行榜数据进行排序时，我们会用到Redis中的有序集合对象（sorted set），跳跃表就是有序集合的底层实现之一。跳跃表的查找速度平均为O(logN)，最差为O(N)。

用跳跃表访问表头及表尾节点的你复杂度均为O(1)，也可以在O(1)时间内得到表中节点的数目。

跳跃表默认会以分值的大小递增排序，分值相同则根据成员对象的字典序进行排序。

6、整数集合（intset）
整数集合是Redis中集合的底层实现之一。

在将元素（整数）加入集合时，不需要考虑类型，因为整数集合中做了升级（upgrade）逻辑：当加入元素到整数集合时，如果新元素的类型比现有的类型要大时，整数集合会进行升级。升级逻辑提高了灵活性以及比较好的节约了内存。

集合中查找元素是通过二分法实现的。

7、压缩列表（ziplist）
压缩列表是一种为了节约内存而开发的顺序数据结构，是列表及哈希对象的底层实现之一。

8、Redis 的数据对象
Redis有五种数据对象：string（字符串），list（列表），hash（哈希），set（集合）及 zsetsorted set（有序集合)。这些对象就是依赖底层数据结构构造的。

9、一个字符串类型的值能存储最大容量是多少。
512M

10、Redis 的持久化机制是什么？各自的优缺点
Redis提供两种持久化机制 RDB 和 AOF 机制

（1）RDB是指用数据集快照的方式半持久化记录 redis 数据库的所有键值对，在某个时间点将数据写入一个临时文件，持久化结束后，用这个临时文件替换上次持久化的文件，达到数据恢复。
（2）AOF（Append-only file）持久化方式，是指所有的命令行记录以 redis 命令格式完全持久化存储保存为 aof 文件。

两者区别：
（1）RDB模式数据安全性低。RDB 是间隔一段时间进行持久化，如果持久化时 redis 发生故障，会发生数据丢失；
（2）AOF 文件比 RDB 文件大，且恢复速度慢；
（3）数据集大的时候，AOF 比 RDB 启动效率低

11、RDB备份操作。
BGSAVE命令下Redis会创建一个子进程来进行RDB文件的创建，而父进程继续处理命令请求，这样就不会导致在RDB文件创建的过程中阻塞其他的命令请求。而SAVE命令是会阻塞命令请求直至完成SAVE命令。因此Redis一般情况下都是执行BGSAVE命令。当Redis服务器在启动时，会加载redis.conf文件，其中保存着RDB文件备份的条件，比如：save 900 1。指的是如果在900s内至少对数据库进行了一次修改则执行BGSAVE命令

12、aof同步策略
1）no：会将aof_buf缓冲区中的所有内容写到AOF文件，但并不同步AOF文件；
2）everysec：每秒同步；
3）always：每一步做同步。

13、AOF的重写。
Redis提供了AOF的重写（rewrite）功能。通过重写，可以创建一个新AOF文件来代替现有的AOF文件，新的AOF文件中国的操作是通过直接读取服务器当前的数据库状态来实现的，所以没有其他浪费空间的冗余命令。AOF重写功能是这样实现的：从现有数据库中读取键现在的值，然后用一条命令去记录键值对，而后代替之前记录这个键值对的多条命令。

AOF后台重写：
在子进程执行AOF重写的期间，服务器还会接收新的请求命令，这些命令可能会对现有数据库状态进行修改，从而让服务器当前状态与重写后的AOF文件保存的状态不一致。为了解决上述这个问题，Redis服务器设置了一个AOF重写缓冲区，此缓冲区在子进程进行重写后使用，当Redis接收一个命令时，会将这个命令发送到AOF重写缓冲区，当重写完成后，只需要让子进程将AOF重写缓冲区中的数据写入新的AOF文件中即可。

14、Redis主从复制
Redis的复制功能分为同步（sync）和命令传播（command propagate）两个步骤

15、Redis的同步机制
Redis可以使用PSYNC指令进行同步，有两种同步方式：完整重同步（full resync）和部分重同步（partial resync）：
1） 完整重同步一般用在第一次复制的时候，通过让主服务器创建并发送RDB文件给从服务器，并向从服务器发送缓冲区内的写命令来实现；
2） 部分重同步一般用于断线重连的复制，当从服务器断线后又重新上线，如果满足一定条件主服务器可以将断线这段时间的写命令发送给从服务器，从而使主从数据库状态保持一致。

15、部分重同步实现
主服务器与从服务器都会维护一个重同步的偏移量（psync_offset）：主服务器想从服务器传播N字节数据，就会将自己的偏移量加N；同理从服务器收到N字节数据就将自己的偏移量加N，从而我们可以根据偏移量来确定从服务器和主服务器的数据库状态是否一致。

16、主服务如何决定使用完整重同步还是部分重同步
主服务器维护了一个长度固定的队列，默认大小为1024*1024字节，即1M。如果从服务器的psync_offset还在复制备份队列范围内，那将执行部分重同步操作；反之将执行完整重同步；另外如果当前连接的主服务器的运行id与之前保存的运行id不一致也不能执行部分重同步。

17、命令传播
一般来说在执行完同步命令后，主从服务器的数据库状态应该保持一致，但是假设在主从服务器刚刚完成同步后，客户端又向主服务器发送了相关的写命令，那么此时主从数据库的状态是不一致的，这个时候主服务器就需要对从服务器执行命令传播：主服务器将执行的写命令发送给从服务器，待从服务器执行后，两者的数据库状态将保持一致。

18、Redis高可用性方案
这里的高可用指当主服务器挂了以后，从服务器应该能担当主服务器的角色，从而提供给用户正常的读写服务。

哨兵（Sentinel）是Redis高可用性的一个解决方案：由一个或多个哨兵实例组成的哨兵系统可以监视多个主服务器及该主服务器下的所有从服务器，当主服务器宕机下线时，哨兵系统可以迅速感知这一情况，从从服务器中选举出一个服务器充当主服务器的角色，进行故障转移

19、哨兵（Sentinel）是什么
哨兵Sentinel本质上是一个运行在特殊模式下的Redis服务器。当Sentinel与一个主服务器或从服务器建立订阅连接后，Sentinel会通过订阅连接以两秒一次的频率向所有被监视的主服务器及从服务器的sentinel:hello频道发生消息。通过这种固定频率的通讯，Sentinel会在通讯时更新各项信息，如主服务器的状态，id以及从服务器的状态，id等。那么通过Sentinel模式，我们就可以实时感知到Redis服务器网络中的情况。

20、哨兵（Sentinel）故障转移方案
遇到主服务器宕机影响服务正常运行的情况，Sentinel给出了以下几个步骤来解决：
1）主观下线：Sentinel会默认以1秒的频率向所有与其创建了命令连接的实例发送PING命令，当实例运作不正常时，Sentinel将会开启这个实例的SRI_S_DOWN标志，从而表示这个实例进入主观下线状态；
2）客观下线：当一个主服务器被Sentinel判断为主观下线后，为了确保其真的下线，该Sentinel会向其他监视着一主服务器的Sentinel进行询问，看其他的Sentinel是否也将其标记为主观下线，若超过一定数量（sentinelRedisInstance结构中的quorum属性）的Sentinel都认为此主服务器下线了，那么该主服务器会被标记为客观下线；
3）选举领导Sentinel：每个Sentinel都有选举权以及被选举权，并且每个发现主服务器客观下线的Sentinel都会要求别的Sentinel将自己设置为老大。但是，在一个配置纪元内，所有的Sentinel都只有一次将某个Sentinel投票为老大的机会，投票的规则是先到先得：最开始向Sentinel A发送投票请求的Sentinel将成为Sentinel A的局部领导，之后所有的投票请求都将被拒绝，因为老大只能设置一次。只有被超过一半的Sentinel选举为局部领导的Sentinel才能成为真正的领导；
4）故障转移：故障转移分为以下几个步骤：1）准备执行故障转移，确认本Sentinel为领导Sentinel；2）选择合适的从服务器作为新的主服务器；3）向被选中的从服务器发送 SLAVEOF no one 命令，将它升级为新的主服务器；4）让其他尚未同步新主服务器的从服务器同步该新主服务器

21、Redis集群
通过Redis集群（cluster）也能起到与哨兵相同的作用。集群通过分片（sharding）来进行数据共享，并提供复制和故障转移的功能。Redis集群数据默认在0数据库。

22、槽（slot）是什么
Redis通过分片来保存数据库中的键值对，所谓分片是指数据库被分为了16384个槽（slot），所有的键值对都在这16384个槽中，每个 key 通过 CRC16 校验后对 16384 取模来决定放置哪个槽，集群中的节点会被指派处理这些槽。

23、Redis集群如何执行命令
当集群上线后，客户端就可以向集群中的节点发送命令请求，当客户端的命令与键相关时，接受命令的节点会计算这个键所在的槽，并且检查自己是否负责这个槽，如果刚好负责这个槽则执行命令，否则节点向客户端返回MOVED错误，并转向正确的节点，并让正确的节点执行命令。

24、Redis集群重分片
可以通过重新分片操作将指派给节点A的槽重新指派给节点B，重分片过程可以在线进行即不需要让集群下线。重分片步骤如下：
1）让目标节点准备从源节点导入属于slot槽的键值对；
2）让源节点将属于槽slot的键值对迁移到目标节点；
3）所有属于槽slot的键值对都迁移到目标节点；
4）将槽slot指给向集群中任意一个节点A，并且这个信息会传播到整个集群，从而让所有的节点都知道槽slot指派给了节点A。

25、槽迁移时的命令处理
如果槽slot正在进行迁移，如果此时客户端的一个命令请求刚好是涉及到slot里的键，源节点首先会在自己的数据库中查找这个键，如果找到的话就直接执行命令，如果没找到，说明这个键已经迁移到目标节点，此时源节点将返回ASK错误，并引导客户端的命令导入正确的节点。

26、集群中的下线检测
集群中每个节点都会定期向其他节点发送PING消息，来检测节点是否在线，如果节点没有在规定时间内返回PONG，那个发送PING的节点就会将此节点标记为疑似下线（probable fail），当一个主节点A知道节点B将C疑似下线时，主节点会将主节点B的下线报告（failure report）加入到clusterNode的fail_reports链表中。当节点被标记为疑似下线PFAIL时，会判断这个节点是否真的下线了。判断逻辑是向集群中其他节点询问是否认为此节点下线，如果半数以上节点认为此节点下线了，那么这个节点就会被标记为下线FAIL。

27、集群中的故障转移
当从节点发现主节点下线时，就会对主节点进行故障转移，步骤总体与哨兵中的逻辑类似。
1）从主节点的所有从节点中选择一个节点成为主节点；
2）新的主节点会将已下线的主节点的槽指派全部取消，并把这些槽全部指派给自己；
3）新主节点会向集群广播一条PONG消息，从而让其他主节点知道这个节点成为了新的主节点；
4）新主节点开始接收处理和自己负责的槽相关的命令请求，故障转移完成

28、Redis数据淘汰策略
volatile-lru：从已设置过期时间的数据集 （server.db[i].expires）中挑选最近最少使用的数据淘汰
volatile-ttl：从已设置过期时间的数据集 （server.db[i].expires）中挑选将要过期的数据淘汰
volatile-random：从已设置过期时间的数据集 （server.db[i].expires）中任意选择数据淘汰
allkeys-lru：从数据集（server.db[i].dict）中挑选最近最少使用的数据淘汰
allkeys-random：从数据集（server.db[i].dict）中任意选择数据淘汰
no-enviction（驱逐）：禁止驱逐数据

29、Redis查找符合某些规则的key
在巨大的数据量的情况下，做类似于查找符合某种规则的Key的信息，这里就有两种方式：
一是keys命令，简单粗暴，由于Redis单线程这一特性，keys命令是以阻塞的方式执行的，keys是以遍历的方式实现的复杂度是 O(n），Redis库中的key越多，查找实现代价越大，产生的阻塞时间越长；
二是scan命令，以非阻塞的方式实现key值的查找，绝大多数情况下是可以替代keys命令的，可选性更强。但是会有一定的重复概率，在客户端做一次去重就可以了。

30、如果有大量的 key 需要设置同一时间过期，一般需要注意什么
如果大量的 key 过期时间设置的过于集中，到过期的那个时间点，redis 可能会出现短暂的卡顿现象。一般需要在时间上加一个随机值，使得过期时间分散一些。

31、Redis分布式锁
先拿 setnx 来争抢锁，抢到之后，再用 expire 给锁加一个过期时间防止锁忘记了释放，这种方案有缺陷，因为是两个操作，不能保证原子性。而且当在 setnx 之后执行 expire之前进程意外crash，将会导致锁一直没有办法被释放。
但是set 指令有非常复杂的参数，这个应该是可以同时把 setnx 和expire 合成一条指令来用

32、Memcache 与 Redis 的区别都有哪些。
（1）存储方式：Memecache 把数据全部存在内存之中，断电后会挂掉，数据不能超过内存大小。 Redis有部份存在硬盘上，这样能保证数据的持久性；
（2）数据支持类型：Memcache 对数据类型支持相对简单。 Redis 有复杂的数据类型；
（3）使用底层模型不同：它们之间底层实现方式以及与客户端之间通信的应用协议不一样。
